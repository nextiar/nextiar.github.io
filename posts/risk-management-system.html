금융사 리스크 관리 시스템 구축기
— 설계부터 운영까지

📅 2025년 4월 15일 ⏱️ 22분 읽기 👤 NEXTIAR

자산운용사 IT 팀장으로 일하던 2019년, 우리 회사는 리스크 관리 시스템(RMS)을 전면 재구축하기로 결정했습니다. 기존 시스템은 엑셀 기반의 반수동 프로세스로, 포지션이 한도를 초과해도 5분 이상 지나야 알 수 있는 구조였습니다. 금융감독원 검사에서 지적을 받기 전에 먼저 고쳐야 했습니다.

"리스크 관리 시스템은 문제가 없을 때는 아무도 주목하지 않지만, 문제가 생기면 모든 것을 설명해야 합니다."

요구사항 분석

리스크 관리 시스템의 요구사항은 크게 세 가지였습니다.

실시간 포지션 집계

여러 자산군(주식, 채권, 파생상품, 외환)에 걸친 포지션을 실시간으로 집계해야 합니다. 기존에는 시스템 간 인터페이스가 배치(batch) 기반이라 딜레이가 발생했습니다.

📋 핵심 요구사항
  • 포지션 업데이트 지연: 1초 이내
  • 한도 초과 시 즉시 알림 (이메일 + SMS + 시스템 팝업)
  • 감사 추적(Audit Trail): 모든 한도 변경 이력 보관
  • 규제 보고서 자동 생성 (금융감독원 양식)

한도 관리

한도는 다층 구조입니다. 회사 전체 → 본부 → 팀 → 펀드 → 트레이더 순으로 계층화되며, 상위 한도를 초과할 수 없습니다.

시스템 아키텍처

이벤트 드리븐 아키텍처를 채택했습니다. 핵심 선택은 Apache Kafka를 이벤트 버스로 사용하는 것이었습니다. 

# 시스템 구성
[OMS] → [Kafka: trade-events] → [Position Aggregator]
[FMS] → [Kafka: fund-events]  →          ↓
[FX]  → [Kafka: fx-events]   → [Risk Engine] → [Alert Service]
                                      ↓
                               [Limit DB (Redis)]
                                      ↓
                               [Reporting Service]

Kafka 토픽 설계

토픽 설계에서 가장 중요한 결정은 파티션 수였습니다. 파티션은 나중에 줄일 수 없으므로 충분히 크게 잡아야 합니다. 

# 토픽 생성 (실제 운영 설정)
kafka-topics.sh --create \
  --topic trade-events \
  --partitions 12 \        # 소비자 수의 3배
  --replication-factor 3 \ # 금융권 최소 복제본 수
  --config retention.ms=604800000 \  # 7일 보관
  --config min.insync.replicas=2

포지션 계산 엔진

포지션 집계는 간단해 보이지만 실제로는 복잡합니다. 환율 변환, 파생상품 델타 계산, 펀드 간 넷팅 처리 등이 필요합니다. 

# 포지션 계산 로직 (Python 의사코드)
def calculate_position(trade_event):
    base_position = get_current_position(trade_event.fund_id,
                                         trade_event.asset_id)

    # 환율 적용 (기준 통화: KRW)
    if trade_event.currency != 'KRW':
        fx_rate = get_realtime_fx(trade_event.currency, 'KRW')
        base_amount = trade_event.amount * fx_rate
    else:
        base_amount = trade_event.amount

    # 파생상품은 델타 기준
    if trade_event.asset_type == 'DERIVATIVE':
        delta = get_delta(trade_event.asset_id)
        adjusted_amount = base_amount * delta
    else:
        adjusted_amount = base_amount

    new_position = base_position + adjusted_amount

    # 한도 체크
    check_limits(trade_event.fund_id, new_position)

    return new_position

한도 체크 성능 최적화

한도 체크는 모든 트레이드마다 실행됩니다. 하루 수십만 건의 체결이 발생하는 환경에서 이 연산이 병목이 되면 안 됩니다.

초기에는 PostgreSQL에 바로 조회했는데, 피크 타임에 응답 시간이 100ms를 넘었습니다. Redis를 L1 캐시로 추가하고 PostgreSQL은 영구 저장소로만 사용하도록 아키텍처를 수정했습니다. 

# Redis 한도 저장 구조
HSET limit:fund:{fund_id} \
  equity_limit 1000000000 \   # 주식 한도 10억
  bond_limit   5000000000 \   # 채권 한도 50억
  fx_limit     500000000  \   # 외환 한도 5억
  total_limit  6000000000     # 전체 한도 60억

# 포지션 저장
HSET position:fund:{fund_id} \
  equity   850000000 \
  bond     2300000000 \
  fx       120000000

# 한도 체크 (Lua 스크립트로 원자성 보장)
local position = redis.call('HGET', KEYS[1], ARGV[1])
local limit = redis.call('HGET', KEYS[2], ARGV[1])
if tonumber(position) > tonumber(limit) then
  return 1  -- 한도 초과
end
return 0

Redis 도입 후 한도 체크 응답 시간이 100ms → 0.8ms로 단축되었습니다.

알림 시스템

한도 초과 알림은 즉각적이어야 하지만, 너무 많은 알림은 알림 피로(Alert Fatigue)를 유발합니다. 다음 규칙으로 알림을 계층화했습니다.

  • 한도 80% 도달: 해당 트레이더에게 이메일 알림
  • 한도 90% 도달: 트레이더 + 팀장에게 이메일 + 시스템 팝업
  • 한도 95% 도달: 전 관련자에게 SMS + 이메일 + 팝업, 추가 주문 발주 경고
  • 한도 100% 초과: 즉시 주문 거부 + CRO(최고리스크관리책임자) 직보

배운 것들

이 프로젝트에서 가장 중요하게 배운 것은 금융 도메인 지식의 중요성입니다. 아무리 뛰어난 엔지니어도 "레그(leg)"가 뭔지, "넷팅(netting)"이 어떻게 작동하는지 모르면 제대로 된 시스템을 만들 수 없습니다.

두 번째는 규제 요건을 처음부터 설계에 반영하는 것입니다. 나중에 감사 기능을 추가하는 것은 처음부터 설계하는 것의 3배 비용이 들었습니다.
