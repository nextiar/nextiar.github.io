IT 운영 고가용성

쿠버네티스로 구현하는
금융 시스템 고가용성
— 실전 운영 가이드

📅 2025년 4월 28일 ⏱️ 18분 읽기 👤 NEXTIAR

2022년, 저는 국내 중견 자산운용사의 핵심 IT 시스템을 전통적인 물리 서버 환경에서 Kubernetes 기반 클라우드 네이티브 환경으로 전환하는 프로젝트를 리딩했습니다. 이 글은 그 과정에서 얻은 시행착오와 실전 설정값을 기록합니다.

⚠️ 읽기 전 참고사항

이 글의 설정값들은 특정 환경(온프레미스 + AWS 하이브리드, 금융권 컴플라이언스 적용)에서 도출된 것입니다. 환경에 따라 튜닝이 필요합니다.

왜 쿠버네티스였나

금융권에서 쿠버네티스는 여전히 조심스러운 선택입니다. 레거시 시스템이 많고, 규제 요건이 엄격하며, "검증된 기술"을 선호하는 문화가 있습니다. 그럼에도 쿠버네티스를 선택한 이유는 명확했습니다.

  • 배포 자동화: 기존에는 배포 하나에 2~3명이 붙어 반나절을 소모했습니다. Helm + ArgoCD로 15분으로 단축
  • 자가 복구: Pod 장애 시 자동 재시작. 주문 처리 시스템의 가용성이 크게 향상
  • 리소스 효율화: 서버 利用率이 20%대에서 60%대로 향상. 인프라 비용 절감
  • 수평 확장: 시장 개장 전후 트래픽 스파이크 대응을 자동화

아키텍처 설계

금융 시스템의 가장 큰 제약은 데이터 주권과 지연 시간(latency)입니다. 주문 처리 시스템의 경우 1ms 지연도 민감한 문제가 됩니다.

클러스터 설계

저희는 다음과 같은 세 개의 클러스터를 운영합니다.

# 클러스터 구성
production-cluster   # 온프레미스, 핵심 거래 시스템
staging-cluster      # AWS, 배포 전 검증
dr-cluster           # AWS, 재해 복구용

노드 풀 구성

금융 시스템의 특성상 워크로드를 격리해야 합니다. 주문 처리 시스템과 리포팅 시스템이 같은 노드를 공유하면 안 됩니다. 

# node-pool 구성 예시
apiVersion: v1
kind: Node
metadata:
  labels:
    workload-type: critical-trading   # 주문 체결용
    workload-type: batch-reporting    # 배치 리포팅용
    workload-type: api-gateway        # API 서버용

고가용성 핵심 설정

99.99% 가용성(연간 다운타임 52분 이하)을 목표로 설정한 핵심 파라미터들입니다.

PodDisruptionBudget

apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: order-system-pdb
spec:
  minAvailable: 2          # 최소 2개 Pod 항상 유지
  selector:
    matchLabels:
      app: order-system

리소스 제한

금융 시스템에서 OOMKilled는 치명적입니다. 리소스 설정을 과하다 싶을 정도로 보수적으로 잡으세요. 

resources:
  requests:
    memory: "2Gi"
    cpu: "1000m"
  limits:
    memory: "4Gi"      # requests의 2배가 안전선
    cpu: "2000m"

HPA 설정

apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: order-system-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: order-system
  minReplicas: 3          # 최소 3개 (가용존 분산)
  maxReplicas: 20
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 60    # 60% CPU 초과 시 스케일아웃
  behavior:
    scaleUp:
      stabilizationWindowSeconds: 30   # 빠른 확장
    scaleDown:
      stabilizationWindowSeconds: 300  # 느린 축소 (장중 보수적 접근)

삽질 기록 — 실패에서 배운 것들

이 섹션이 가장 솔직한 부분입니다. 교과서에 없는 실패들을 기록합니다.

실패 1: 장 개시 직전 스케일아웃 타이밍 문제. HPA가 트래픽이 몰리는 오전 8시 50분에 반응하기 시작했는데, Pod 기동 시간(Cold Start)이 30초~1분 걸려서 결국 9시 장 개시 직후 응답 지연이 발생했습니다. 해결책: CronJob으로 8시 40분에 미리 스케일아웃.

실패 2: etcd 디스크 I/O 병목. 온프레미스 환경에서 etcd를 일반 HDD에 올렸더니 클러스터 전체가 불안정해졌습니다. NVMe SSD로 교체 후 안정화. etcd는 반드시 SSD에 올리세요.

실패 3: 네트워크 정책 미설정으로 인한 컴플라이언스 지적. 금융감독원 감사에서 Pod 간 통신이 무제한으로 허용된 것이 지적됐습니다. NetworkPolicy로 최소 권한 통신을 적용하는 데 2주가 걸렸습니다.

결과

전환 후 6개월간의 결과입니다:

  • 가용성: 99.2% → 99.97% 향상
  • 배포 시간: 4시간 → 15분 단축
  • 인프라 비용: 약 28% 절감
  • 장애 복구 시간(MTTR): 평균 45분 → 8분 단축
